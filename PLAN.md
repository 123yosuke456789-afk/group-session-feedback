# グループセッション・フィードバックツール — 全体プラン

## 最終目的（ゴール）

**オンラインスクールのグループセッションで、各自が自分の作品を共有し、フィードバックをもらう。**

- 口頭だけだと「どの部分の話か」が伝わりにくい
- だから **「作品の特定の部分を指して、そこにコメントを書ける」** システムをつくる
- 参加者全員が、同じ作品を見ながら、**場所を指定して自由にコメント**できるようにする

---

## いまの状態を一言で

- **動くもの**はもうある（あなたのPCで `npm run dev` すると使える）
- ただし **あなたのPCが起動しているときだけ** 使える
- データ（セッション・コメント）は **メモリだけ** なので、サーバーを止めると消える

**「みんなが、あなたのPCに依存せず、いつでも使える」** にするのが、このプランの本番ゴールです。

---

## 用語の整理（理解を追いつかせるため）

| 用語 | 意味（ざっくり） |
|------|------------------|
| **ローカル / localhost** | あなたのパソコンの中だけで動いている状態。外からは見えない。 |
| **デプロイ** | 作ったアプリを「いつでも動いているサーバー」に置くこと。あなたのPCを消しても動き続ける。 |
| **Vercel** | 今回使う「サーバーを貸してくれるサービス」。Next.js と相性がよく、無料枠でデプロイできる。あなたのコードを預けると、**あなたのPCとは別の、常時起動しているコンピュータ**で動かしてくれる。 |
| **実験** | 少人数（例：あなた＋母さん）で、動作や使い心地を試す段階。 |
| **本番** | オンラインスクールの仲間全員に「いつでも使ってね」と渡して使う段階。 |

---

## 進め方の全体像（3段階）

```
[Phase 0] いまここ
     ↓ 実験で「使える」ことを確認
[Phase 1] Vercel にデプロイ → 誰でも・いつでもアクセス可能に
     ↓ 必要に応じて
[Phase 2] データを残す・認証する（オプション）
```

---

## Phase 0：実験（いま〜ここまで）

**目的:** 「場所を指定してコメントする」が本当に使えるか、少人数で試す。

- **やること**
  - あなたのPCで `npm run dev` を起動したまま使う
  - 母さんなど、**1人でもいいので**試してもらう
    - **同じ Wi‑Fi（同じ家）なら:** ターミナルに表示される `http://192.168.x.x:3000` を共有するだけでよい。**ngrok は不要。**
    - 別の場所（別ネットワーク）の人と実験したい場合だけ、ngrok で一時的にURLを公開する方法がある（必須ではない）
  - 実際に「作品URLを入れる → 招待リンクを送る → 相手がピンでコメントする」をやってみる
- **確認したいこと**
  - 作品のURLが iframe で表示できるか（表示できないサイトはスクショ画像URLや Figma などで代用）
  - ピンとコメントの流れが、口頭より伝わりやすいか
- **理解のポイント**
  - この段階では「あなたのPCが電源オン＋`npm run dev` のときだけ」動いている
  - **ngrok は本番では使わない。** テストも、同じ Wi‑Fi なら不要。

**終了の目安:** 母さん（または誰か1人）と、1回でもフィードバックのやりとりができた。

**具体的な手順:** [docs/phase0-テスト手順.md](./docs/phase0-テスト手順.md) にステップごとのやり方をまとめています。

---

## Phase 1：本番の土台 — Vercel にデプロイ（勧めたいメインの一歩）

**目的:** あなたのパソコンの起動に関係なく、**みんながいつでも同じURLで使える**ようにする。

- **やること**
  1. **Vercel のアカウント**を作る（GitHub やメールで登録、無料）
  2. **Git** でこのプロジェクトを GitHub に上げる（まだなら）
  3. Vercel の画面で「この GitHub リポジトリからデプロイ」を選ぶ
  4. 数分で **本番URL**（例: `https://グループセッション.vercel.app`）が発行される
  5. そのURLをオンラインスクールの仲間に共有する
- **あなたのPC**
  - デプロイ後は **このツールのためにPCを起動しておく必要はない**
  - コードを直したら、Git に push すると Vercel が自動で再デプロイする（設定すれば）
- **データについて（Phase 1 のまま）**
  - セッション・コメントは **いまのままメモリ**（Vercel のサーバー再起動で消える）
  - スクールの1回のセッション（1〜2時間）の間は通常問題にならない
  - 「過去のセッションをずっと残したい」なら Phase 2 でデータベースを検討

**理解のポイント**
- **Vercel = 「あなたのコードを動かす、常時起動のコンピュータ」を貸してくれる場所**
- あなたのPCは「開発用」、Vercel は「みんなに使ってもらう用」と分かれる

**終了の目安:** スクールの仲間に「このURLで、いつでも使ってね」と渡し、1回のグループセッションで実際に使えた。

---

## Phase 2：必要なら「残す・守る」（オプション）

**目的:** 最終目的は変わらない。「過去のコメントを残したい」「参加者を限定したい」が出てきたときの拡張。

- **データを残す**
  - 今: メモリだけ → 再起動で消える
  - 拡張: **データベース**（例: Vercel Postgres, Supabase）を用意し、セッション・コメントを保存する
  - こうすると「先週のセッションのコメントをあとから見返す」ができる
- **参加者を限定する**
  - 今: 招待リンクを知っている人は誰でも参加できる
  - 拡張: **参加コード**や**ログイン**を入れて、スクールの受講生だけが使えるようにする
- **タイミング**
  - Phase 1 でしばらく使ってみて、「コメントを残したい」「部外者には見せたくない」という要望が出てからでよい

---

## 進め方のオススメ（実験→本番の流れ）

1. **まず Phase 0 を小さく完了する**
   - 母さん（または1人）と、別の場所から1回だけでもいいので「作品共有 → ピンでコメント」を試す
   - ここで「やっぱりここが使いにくい」などが分かれば、デプロイ前に直せる
2. **問題なければ Phase 1 に進む**
   - GitHub にコードを上げる → Vercel でデプロイ → 本番URLを取得
   - スクールのグループに「このURLでフィードバック用ツール使えます」と案内する
3. **運用しながら Phase 2 を検討**
   - 「コメントを残したい」「参加者を制限したい」が出てきたら、そのタイミングでデータベースや認証を検討する

---

## まとめ

| 段階 | 目的 | 誰が・いつ使えるか |
|------|------|---------------------|
| **Phase 0（実験）** | 少人数で「場所指定コメント」を試す | あなたのPC起動中。同じ Wi‑Fi なら `192.168.x.x:3000` を共有するだけ（**ngrok 不要**） |
| **Phase 1（本番の土台）** | みんながいつでも使えるようにする | **Vercel にデプロイ** → あなたのPCに依存しない・常時アクセス可能（**ngrok 不要**） |
| **Phase 2（オプション）** | データを残す・参加者を制限する | 必要になったら DB・認証を追加 |

**ngrok について:** テストは同じ Wi‑Fi で十分なので **基本的に不要**。本番は Vercel のURLだけで完結するので **不要**。

**最終目的**は変わらない:  
**オンラインスクールのグループセッションで、各自が作品を共有し、特定の部分を指して自由にコメントできるシステム。**

まずは Phase 0 で一度「別の場所の人と使えた」ことを確認し、そのあと **Phase 1（Vercel デプロイ）** に進むのが、いちばん無理のない進め方です。
